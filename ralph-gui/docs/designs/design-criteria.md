# Ralph Workflow - Design Criteria

This document defines the binding design standards for all visual and interaction
work on Ralph Workflow. It complements the UX Principles document — that document
covers *how the experience should feel*, this one covers *how the design should
be executed*.

---

## Required Tooling

**The `frontend-design` skill must be used for all UI work.**

Every screen, component, or visual change must be designed and reviewed through
the `frontend-design` skill. This is not optional. The skill exists to prevent
generic, template-looking interfaces and ensure every piece of UI is
intentionally crafted.

---

## Core Design Principles (from frontend-design)

### Distinctive Over Generic
- Every screen should have a clear visual identity — it should look like *this*
  product, not like a default Material scaffold
- Avoid the "AI-generated look" — generic card grids, uniform spacing everywhere,
  no personality. If a design could belong to any app, it's not specific enough
- Design decisions should be deliberate and traceable to a reason, not just "it's
  the default"

### Hierarchy Through Contrast
- Establish visual hierarchy through size, weight, color, and spacing — not
  through decoration
- The most important element on every screen should be immediately obvious
- Use contrast to guide attention: high contrast for primary content, low contrast
  for supporting information
- Don't rely on borders and dividers when spacing and background shifts can do
  the same job more cleanly

### Restraint Over Excess
- Use color sparingly — the amber accent is powerful *because* it's rare. If
  everything is amber, nothing is emphasized
- Limit visual noise: fewer borders, fewer shadows, fewer competing elements
- White space (or in our case, dark space) is a design tool, not wasted space
- Every visual element must earn its place — if removing it doesn't hurt
  comprehension, remove it

### Density Appropriate to Context
- Developer tools warrant higher information density than consumer apps — don't
  over-space things
- But density doesn't mean clutter. Dense and organized is the goal; dense and
  chaotic is the failure mode
- Different views warrant different density: the dashboard is medium-density
  overview, the log viewer is high-density monitoring, the wizard is low-density
  focused input

### Dark Theme as a First-Class Design
- Dark mode is not "invert the colors." It requires its own careful palette
  with proper surface layering
- Surfaces are distinguished by subtle lightness shifts, not by borders or
  shadows alone
- Text must be deliberately chosen for each surface level — the same white
  doesn't work on every dark surface
- Elevation in dark mode is communicated through lighter surfaces, not darker
  shadows

## Branding & Logo

### Source Asset
The Ralph Workflow logo lives in `vendor/ralph-workflow-brand/assets/logo.svg`. This is
the single source of truth for all logo and icon usage.

The logo is an SVG using `currentColor`, meaning it adapts to whatever color
context it's placed in. The design features a stylized "R" within a rounded
rectangle, with a small chevron detail.

### Usage Within the App
- The logo must appear in the title bar or application header area
- The logo must be used as the activity bar brand mark (top of the sidebar)
- The logo should appear in the onboarding/welcome screen
- The logo should appear in empty states where appropriate (e.g., no workspaces open)
- When used as an icon within the app, render it using the amber accent color
  (`#f59e0b`) on dark surfaces, or the primary text color (`#f0f0f2`) when
  amber would conflict with status colors
- Do not distort, rotate, or add effects to the logo
- Maintain clear space around the logo — at minimum, the width of the chevron
  detail on all sides

### Distribution & Platform Icons
The branding tool in `vendor/ralph-workflow-brand` generates platform-specific
icons from the source SVG. These must be used for:

- **macOS:** App icon (`.icns`) — generated via `--platform macos`
- **Windows:** App icon (`.ico`) — generated via `--platform windows`
- **Linux:** Desktop entry icons (multiple PNG sizes) — generated via `--platform linux`
- **Tauri config:** Reference the generated icons in `tauri.conf.json`

To generate all platform icons:
```
cd vendor/ralph-workflow-brand
cargo run -- --output-dir ../../ralph-gui/src-tauri/icons --color "#f59e0b" --platform all
```

The brand color for distribution icons is amber `#f59e0b` — matching the
app's accent color. This ensures the app icon is recognizable and consistent
with the in-app experience.

Do not use placeholder icons, framework default icons, or hand-made
approximations. Always generate from the source SVG.

### Favicon / Web Context
If the app ever exposes a web view or local dev server, use the web assets
generated by the branding tool (`favicon.svg`, `favicon.ico`,
`apple-touch-icon.png`).

The full design system (color tokens, typography scale, spacing grid, elevation,

## Design Review Questions

Before any UI work is considered complete, answer these:

1. **Is the `frontend-design` skill being used?** If not, stop and use it.
2. **Does this look like Ralph Workflow, or does it look like a template?** Check that
   the design reflects the product's identity, not a generic scaffold.
3. **Is the logo used correctly?** Source SVG, proper color, clear space, no
   distortion.
4. **Are distribution icons generated from the brand tool?** No placeholder
   or default Tauri icons.
5. **Is the visual hierarchy clear?** Can you identify the primary, secondary,
   and tertiary elements without reading the content?
6. **Is the dark theme well-executed?** Proper surface layering, appropriate
   contrast, no "inverted light mode" shortcuts?
7. **Is density appropriate?** Not wastefully sparse, not chaotically dense —
   matched to the view's purpose?
8. **Is color used with restraint?** Amber is accent, not wallpaper. Status
   colors are reserved for status.
