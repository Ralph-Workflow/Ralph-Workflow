## [Unreleased]

### Changed

- **refactor(pipeline): route plumbing-direct-call fallback through `DefaultPipelineFactory` and add `pro_hooks` plumbing** — `ralph/pipeline/plumbing/commit_plumbing.py` and `ralph/pipeline/plumbing/smoke_plumbing.py` now construct their fallback `PipelineDeps` via `DefaultPipelineFactory().build(...)` instead of calling `build_default_pipeline_deps(...)` directly. Both `run_commit_plumbing` and `run_smoke_plumbing` accept a new keyword-only `pro_hooks: ProPipelineHooks | None = None` parameter that is forwarded into the factory, so a Ralph Workflow Pro subclassed or replaced factory is honored on the plumbing-direct-call path as well as the main pipeline. The shared composition contract is pinned by new black-box tests in `tests/test_pipeline_factory_default.py` and by the updated seam in `tests/integration/test_plumbing_shared_deps.py`; `tests/integration/test_plumbing_pipeline_core.py` continues to cover the `PipelineCore`-first branch.
- **refactor(pipeline): add `DefaultPipelineFactory` as the single composition root for the main pipeline and plumbing** — `ralph/pipeline/factory.py` now exports a `DefaultPipelineFactory` class that implements the existing `PipelineFactory` Protocol. The factory's `build` method is a thin wrapper over `build_default_pipeline_deps` and applies all 13 `ProPipelineHooks` fields. The five existing call sites (ralph/cli/commands/commit.py, ralph/cli/commands/smoke.py, ralph/cli/commands/run.py, ralph/pipeline/runner.py, ralph/pipeline/parallel/worker_runtime.py) now route through the factory, so the 8 extended Pro DI seams (`policy_bundle`, `policy_bundle_factory`, `registry_factory`, `state_factory`, `recovery_controller_factory`, `marker_watcher_factory`, `snapshot_registry`, `recovery_sleep`) reach plumbing for the first time. The CLI plumbing surfaces now pass `pipeline_deps=deps` to `run_commit_plumbing` / `run_smoke_plumbing` instead of `pipeline_core=core, bridge_factory=build_session_bridge`, using branch 1 of the plumbing dispatch. The test-patch surface in tests/test_cli_commit_command.py, tests/test_cli_smoke.py, and tests/test_parallel_worker_runtime.py (5 worker_runtime patch sites at lines 201, 310, 435, 530, 697-698) was migrated from patching `build_minimal_pipeline_core` / `apply_pro_hooks_to_core` / `build_default_pipeline_deps` on the CLI and worker_runtime modules to patching `DefaultPipelineFactory` instead. New black-box tests in tests/test_pipeline_factory_default.py and tests/integration/test_cli_plumbing_uses_factory.py pin the Protocol compliance, the modular Pro hook propagation, the extended Pro hook propagation (all 8 fields), and the CLI integration. Documented in docs/agents/pro-contract.md as a forward-looking engine capability pending contract amendment; Pro can subclass or replace the factory to inject a custom composition root.
- **feat(idle-watchdog): weight workspace file changes by class** the default ``WorkspaceMonitor`` now drops log / cache / artifact / other file changes from the ``NO_OUTPUT_DEADLINE`` evidence channel; only source-code changes count by default. Source code is valued MUCH higher than log/cache/artifact/other changes when deciding whether workspace activity should defer the watchdog's idle verdict. The five ``WorkspaceChangeKind`` values are: ``source`` (source code / documentation), ``log`` (``*.log`` / ``*.tmp`` / ``*.bak`` / ``*.swp`` / ``*~`` / ``*.pyc`` / ``*.pyo``), ``cache`` (``.git`` / ``__pycache__`` / ``.pytest_cache`` / ``.mypy_cache`` / ``.ruff_cache`` / ``node_modules`` / ``.venv`` / ``.agent/tmp`` / ``.agent/raw`` / ``completion_seen_*.json``), ``artifact`` (``.agent/artifacts``), and ``other`` (anything else). **Behavior change:** the default conservative policy means operators who relied on log-file activity to defer the verdict must opt in by adding ``agent_workspace_change_weights = { log = 1.0, source = 1.0, cache = 0.0, artifact = 0.0, other = 0.0 }`` to the ``[general]`` section of ``ralph-workflow.toml``. See ``docs/agents/timeout-policy.md`` for the full migration note and worked example.
- **fix(interrupt): rename `bridge_pids` kwarg to `bridge_pgids`** on `InterruptController.force_interrupt`, `InterruptController.force_exit`, `InterruptDispatcher.force_exit`, and the second-SIGINT routing in `ralph.interrupt.asyncio_bridge`. The parameter name now matches the value semantics: callers pass PGIDs because the production code routes the values to `kill_process_group(pgid, signal.SIGKILL)`. The legacy `bridge_pids=` keyword is accepted via a deprecation shim and emits a single loguru warning; it forwards to `bridge_pgids`. **`INTERRUPT_EXIT_CODE` (130) is unchanged.**
- **Parallel plan execution is now delegated to the executing AI agent.** The bundled `pipeline.toml` development phase ships with `dispatch_mode = "agent_subagents"`, so the executing agent is the actor that dispatches its own sub-agents per the plan's `work_units` / `parallel_plan` (see the `## Agent-Driven Parallel Execution` block in `planning.jinja` and the `### 9. PARALLEL EXECUTION (AGENT-DRIVEN)` rubric in `planning_analysis.jinja`). **Ralph-managed fan-out is dormant in this build** (not removed): the same-workspace worker machinery is retained in policy for future re-arming, and the bundled default does not use it. To opt back in, set `[phases.development.parallelization] dispatch_mode = "ralph_fan_out"` (see `docs/sphinx/parallel-mode.md` and `docs/sphinx/advanced-pipeline-configuration.md`). For capable agents, the agent's native sub-agent / task capability is enabled by default via `[agents.<name>] subagent_capability = true` in `ralph-workflow.toml` (Claude is the canonical case; other agents leave the value unset and the agent decides at runtime). The new `ralph.testing.audit_parallelization_dormant` audit is wired into `make verify` and enforces 8 invariants across `planning.jinja`, `developer_iteration_continuation.jinja`, `format_docs/plan.md`, `effect_router.py`, `pipeline.toml`, `planning_analysis.jinja`, `docs/sphinx/configuration.md`, and `docs/sphinx/advanced-pipeline-configuration.md`. The continuation template now mirrors the regular `developer_iteration.jinja` template for the `## PARALLEL EXECUTION` block, locked in by `tests/test_continuation_template_parallel_guidance.py` (black-box source-text checks) and a new regression test in `tests/test_audit_parallelization_dormant.py`. Wording follows the audit and the `effect_router.py` WARNING ("dormant", not "removed") so the three sources stay in lockstep.

- **fix(interrupt): add SYNC-path long-running-body test pin, drop dead `grace_period_s` parameter, refactor `__init__.py` re-export, extract `run_shutdown_block` seam, and pin interrupt constants via import-time invariants** — five small changes that close the remaining interrupt-architecture gaps and harden the subsystem against silent regressions. See ADR-0001 D7 and D8.
- **fix(interrupt): production `run_shutdown_block` now uses liveness-based `_wait_for_list_active_empty` (via `begin_interrupt(block=True)`) instead of CPU-based `run_early_escalation_poll`** — closes the "broken at times when the task is long running" bug for I/O-bound agents (writing checkpoints, releasing locks, draining queues) that are alive but have zero CPU usage. The previous production seam spawned a daemon thread that called `run_early_escalation_poll`; the CPU-progress detector's strict `!=` comparison (`_records_show_no_progress`) returned `True` after two consecutive 0-CPU polls and SIGKILLed the agent prematurely. The new production seam is a single `dispatcher.begin_interrupt(grace_period_s=grace_period_s, block=True)` call — no daemon thread, no `threading.Thread.join` — which routes through the dispatcher's liveness-based `_wait_for_list_active_empty` and gives the agent the full `grace_period_s` to die naturally; only an actually-stuck agent (still alive after the grace deadline elapses) is escalated via `force_exit`. The public `InterruptDispatcher.run_early_escalation_poll` method is kept on the dispatcher for backward compatibility with its dedicated tests (`test_early_escalation_poll_kills_when_no_cpu_progress_within_budget`, `test_early_escalation_poll_does_not_kill_when_cpu_progresses`, `test_early_escalation_poll_exits_when_process_dies`) but is NOT wired into the production seam. The `join_timeout_s` parameter on `run_shutdown_block` is kept (defaulted) for backward compatibility but is now UNUSED — the dispatcher's `begin_interrupt(block=True)` waits via the `clock`/`sleep` seams. The 4 contract tests in `test_runner_interrupt.py` and `test_interrupt_dispatcher.py` (the call-shape test for the first-SIGINT production path and the 3 `run_early_escalation_poll` docstring updates) and the asyncio bridge's `test_asyncio_bridge_first_sigint_passes_block_true_via_shutdown_block` test pin the NEW contract. The new black-box regression test `test_run_shutdown_block_does_not_sigkill_alive_but_zero_cpu_long_running_agent` in `tests/test_interrupt_dispatcher.py` proves the alive-but-zero-CPU agent is NOT SIGKILLed under the liveness-based seam (uses canonical `FakeProcessManager` + `_FakeClock` + `_patch_psutil` patterns, runs in < 100ms). The new `_DrainOnBeginDispatcher` wrapper in `test_runner_interrupt.py` drains the records on the first `begin_interrupt` call and re-adds them after `begin_interrupt` returns so the second-SIGINT force-kill handler test (`test_handle_keyboard_interrupt_second_sigint_force_exit_uses_active_pgids`) can verify the `bridge_pgids` contract without the records being killed by the first body. See ADR-0001 D7 and D8.

### Added

- **feat(idle-watchdog): per-kind workspace event breakdown in the NO_OUTPUT_DEADLINE fire diagnostic** the watchdog's ``last_evidence_summary`` workspace channel now carries a per-kind ``kind_breakdown`` (e.g. ``{"source": 5, "log": 0}``) and the ``NO_OUTPUT_DEADLINE`` fire log embeds the full per-channel evidence summary (including the kind breakdown) in the loguru ``extra=`` dict so post-mortem readers can see exactly which workspace kinds were active at the moment of the fire. The ``ChannelEvidenceSummary.kind_breakdown`` field is ``Optional[dict[str, int]]`` and is omitted from ``to_dict()`` when ``None`` for backward-compat.
- **feat(idle-watchdog): new ``[general] agent_workspace_change_weights`` configuration key** for per-kind activity weighting. The default conservative policy is ``{ source = 1.0, log = 0.0, cache = 0.0, artifact = 0.0, other = 0.0 }``; operators can opt a kind in by overriding the value. See ``docs/agents/timeout-policy.md`` for the worked example and the migration note.
- **feat(idle-watchdog): activity-aware idle verdict** — the watchdog now judges a session from all four observable evidence channels, not stdout alone. The four channels are `stdout` (the baseline), `mcp_tool` (Ralph Workflow MCP tool calls / completions), `subagent` (delegated child progress / tool calls), and `workspace` (file-change events from `WorkspaceMonitor`). While any non-stdout channel is fresher than the new `agent_idle_activity_evidence_ttl_seconds` knob (under `[general]`, default `30.0`), the `NO_OUTPUT_DEADLINE` fire is deferred and the watchdog returns `CONTINUE`. Every HARD_STOP diagnostic now embeds a per-channel `evidence_summary` (channel, `last_at`, `age_seconds`, counter) plus an `active_channel` label so an on-call operator or post-mortem reader can see exactly which channels were fresh and which were stale. The absolute `SESSION_CEILING_EXCEEDED` and `CHILDREN_PERSIST_TOO_LONG` ceilings are checked BEFORE the deferral and remain absolute — activity cannot reset either ceiling. The `mcp_tool` channel also covers upstream MCP tool calls proxied through `UpstreamProxyHandler` (`ralph-workflow/ralph/mcp/tools/bridge/_upstream_proxy_handler.py`); a delegated call to an upstream tool is recorded as activity just like a Ralph tool call, so operators who rely on upstream MCP servers for sub-agent work no longer need to disable the activity-aware verdict. Set `agent_idle_activity_evidence_ttl_seconds = 0.0` to opt out and restore the legacy stdout-only behaviour. 9 black-box tests in `tests/agents/test_idle_watchdog_3.py` lock the contract in place under the 60 s combined test budget.
- **feat(mcp): add `unsafe_mode` workflow flag to merge Ralph MCP into agent-native MCP configs** — introduces a new `[general.workflow].unsafe_mode` setting (mirrored by the `--unsafe-mode` CLI flag) that, when enabled, makes Ralph merge its MCP server into each agent's existing MCP configuration instead of overwriting it, giving the agent access to Ralph tools alongside whatever MCP servers it already had. - **Raw-overflow reference managed by `PlainLogRenderer`.** The `[see .agent/raw/<unit>.log]` suffix is now appended by the renderer (`condensed_ref` param on `emit_activity_line`) rather than embedded in the condenser output. `condense_content` called without `overflow_ref` now emits `(truncated)` instead of `(truncated, see raw unavailable)`.
- **Long-content summary is now default-on.** The `↳ summary:` headline line is shown for any content block exceeding 4000 display cells without needing to set an env variable. Set `RALPH_LONG_CONTENT_SUMMARY=0` (or `false`/`no`/`off`) to disable. The summary now uses a sentence-aware extractor (first sentence terminated by `.`, `!`, `?`, or newline), with a 200-character cap for inline summaries and 120 characters for streaming end-line summaries.
- **Streaming `*-end` lines now report fragment count and total char length.** The end-line for a streaming block reads `({n} fragments, {chars} chars)` followed by the headline summary, giving the reader a quick measure of how much content was streamed.
- **Streaming continue tags now carry a 1-based sequence number.** `[content-continue]` is now emitted as `[content-continue#2]`, `[content-continue#3]`, etc. so readers can track the progression of a streaming block.
- **`WARN META [progress]` for dropped ring-buffer lines.** The debounced dropped-line notification is now emitted at `WARN` level (not `INFO`) to match its documented severity and the changelog entry below.

- **Preflight guards unedited starter PROMPT.md.** When running `ralph` with a `PROMPT.md` that still contains the `<!-- ralph:starter-prompt ... -->` marker emitted by `ralph --init`, preflight now fails fast with an error that tells the user to edit the file first, preventing an accidental pipeline run against the example task.
- **`--init` docs now use the real contract.** Quick-start examples now use bare `ralph --init`, compatibility labels remain deprecated/ignored, and rerun output no longer prints a fake template selector.
- **Removed hidden `isolation_mode` compatibility plumbing.** The deprecated `--no-isolation` path is now gone end to end: CLI override input no longer accepts it, config overrides no longer synthesize it, and tests/default guidance only cover the supported execution flags.

### Migration
- Users relying on the previously silent default behavior should now pass `--quiet` (or `-q`) explicitly.
- Users currently passing `--verbosity normal` will see verbose output; they will need to switch to `--quiet` to restore the minimal variant.
- Log parsers that assumed a fixed level set `{INFO,WARN,ERROR,SUCCESS}` must now also accept `MILESTONE` on phase-transition rows.
- Log parsers that matched `[content-continue]` or `[thinking-continue]` exactly must update to match `[content-continue#N]` / `[thinking-continue#N]` (where N is a 1-based integer).
- Tooling that expected dropped-buffer notifications at `INFO` level must update to `WARN`.

### Fixed
- **OS-descendant-only child evidence now fires at the no-progress waiting ceiling.** When an OpenCode agent enters `WAITING_ON_CHILD` with only OS-level process descendants (no fresh scoped label or registry evidence, `alive_by=os_descendant_only_stale_progress`), the idle watchdog now fires at `max_waiting_on_child_no_progress_seconds` (default 600s) instead of the full `max_waiting_on_child_seconds` ceiling (default 1800s). Previously, `SUSPECTED_FROZEN` would fire correctly at 600s but the watchdog would continue waiting until 1800s, keeping a stuck agent alive 20 minutes longer than intended. The effective ceiling is also surfaced correctly in `PROGRESS` and `HARD_STOP` waiting events so log monitors see the true bound.
- **Policy validation now rejects pipelines that declare a tracked budget counter without any matching `post_commit_routes` entry.** Previously, a commit-role phase with `tracks_budget = true` but no matching route would silently fall through to `on_success` instead of raising an error. `validate_policy_completeness` now fails closed with: `phases.<phase>: role='commit' tracks budget counter '<counter>' but no post_commit_routes apply to this phase.`
- **OpenCode false-positive retry bug.** When OpenCode foreground exits rc=0 and (a) child agents are still running OR (b) no children are visible at the exact moment of exit but background work is still pending, `_check_process_result` now waits before declaring resumable failure. Two windows govern the wait: a mandatory `parent_exit_grace_seconds` (default 5s) that polls for late-appearing completion artifacts, `explicit_complete` markers, or newly-registered child agents — covering the race where MCP-driven background subagents have been launched but not yet registered with the ProcessManager — and the existing `descendant_wait_timeout_seconds` (default 30s) that engages once children are visible. Previously, when the foreground exited and no children were visible at that exact instant (a common case for late-spawning MCP subagents and async background work), `OpenCodeResumableExitError` was raised within ~1ms, killing the OpenCode parent before background work could complete. The new grace window eliminates this false positive while preserving the fast path (`TERMINAL_COMPLETE` when completion signals are present at exit time).
- `-D` / `--developer-iters` and `-R` / `--reviewer-reviews` CLI flags now correctly control the number of dev/review cycles. Previously, the pipeline ran exactly one dev cycle and one review pass regardless of the flag values. The `developer_iters` setting controls dev cycles; `reviewer_reviews` controls review cycles.
- `-R=0` now skips the review phase entirely instead of running one forced review pass.
- Analysis loopbacks from `development_analysis` no longer consume a development budget slot. Loopbacks are retries of the same iteration, not new iterations.
- `make typecheck` now runs mypy through `uv run python -m mypy` so it uses the project's virtualenv instead of a system-wide mypy that cannot see project dependencies.

### Removed
- `commit_sha` field removed from `WorkerCompletedEvent`, `WorkerState`, and `WorkerSnapshot`. This was a worktree-era relic always set to empty string in same-workspace v1 mode. Checkpoints from earlier versions load cleanly due to `extra="ignore"` on `WorkerState`.
- `max_dev_continuations` config field (previously declared but never used).
- `_legacy_handle_agent_success` reducer path (unreachable in production — policy is always loaded).
- Unused `developer_iters_option` / `reviewer_reviews_option` decorator stubs in `cli/options.py`.

### Migration Notes
- **Users resuming from a checkpoint created by a prior buggy version**: the stored `development_budget_remaining` and `review_budget_remaining` fields will be `0`, causing the fix to silently repeat the old broken behavior. To get the fix, delete the checkpoint before resuming:
  ```bash
  rm .agent/checkpoint.json
  ```
  Then re-run `ralph -D N -R M` from scratch. Affected users already had a broken run (dev ran once), so this is equivalent to restarting the intended work.
