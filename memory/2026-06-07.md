## GitHub ↔ Codeberg divergence — ROOT CAUSE FIX

### The problem
The sync script `sync_to_github.sh` had a design flaw that created **perpetual oscillation**:
1. Push Codeberg's commits to GitHub
2. THEN overwrite GitHub's README with a mirror notice (separate commit)
3. Next run: GitHub is 1 commit AHEAD of Codeberg → force-pushes Codeberg state (removing mirror notice)
4. THEN overwrites README again (re-creating the mirror notice)
5. ⇄ Infinite loop

This meant every ~30 min sync cycle created 2 pushes and the repos always looked "diverged."

### The fix
Rewrote `sync_to_github.sh` to apply the mirror README BEFORE the single push:
- One atomic push from the sync clone: `[Codeberg HEAD + mirror notice commit]`
- Content-based check: if Codeberg hasn't moved AND the mirror notice on GitHub is already correct → skip entirely
- Result: usually 0 pushes per cycle (just a log line saying "nothing to do")
- When Codeberg advances: 1 push (not 2)
- No oscillation

### Also cleaned up
- Removed the stale editable checkout at `repos/Ralph-Workflow/github-mirror` (was dirty with uncommitted changes, was never the actual sync path)
- Disabled the redundant Python `github_mirror_sync.py` (renamed to `.disabled`) — the bash script is the live sync
- Updated both clones: the gateway-owned mirror (temp sync dir) and the main `/home/mistlight/Ralph-Workflow` clone

### Sync cron
The Gateway cron job `codeberg-github-mirror-sync` (id: b7d74fa6) runs every 30 min and calls `sync_to_github.sh`. That's the only sync mechanism.

The script had:
1. A post-sync step that overwrote GitHub's README with a mirror notice (different from Codeberg)

3. An SEO cannibalization watchdog (OS cron, Sunday 07:15) that checked for the mirror notice and would recreate it if missing

- Disabled `seo_cannibalization_watchdog.py` — renamed to `.disabled`, removed from OS crontab
- Removed dead OS cron entries (`git-sync.sh` that referenced a nonexistent script)
- Verified: both repos now show 0 ahead, 0 behind, identical READMEs
