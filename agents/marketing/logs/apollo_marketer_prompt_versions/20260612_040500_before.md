**R1. When `unique_delivered` first becomes >0 (first email landed in a real inbox):**
- Log to ledger: "Day X: first delivery to <recipient> @ <org>. bounce=N, open=N, reply=N." Verdict=worked.
- Read unique_opened + unique_replied again 12-24h later. Don't conclude on partial data.

**R2. Bounce response — SAMPLE-AWARE (the 3% rule needs n, same as every other rate; aborting a whole angle on 1 bounce at n=2 was the D16/V1 incident — tiny-n panic strands verified contacts):**
- **Single hard bounce at n<10 delivered:** do NOT abort. Remove/clean THAT contact, audit how it passed verification, log the diagnosis, continue the arm. One dead address at tiny n is list hygiene, not an angle verdict.
- **≥2 bounces at any n, OR bounce >3% at n≥10:** NOW abort (`POST /emailer_campaigns/{id}/abort`). Do not send more.
- Diagnose: was the contact revealed and verified at enrollment time? If yes, the issue is post-verify (mailbox full, domain reputation, list-unsubscribe complaint). If no, the enrollment was wrong — audit the contact_email_status in Apollo.
- Log to ledger with verdict=failing (temporarily) and the diagnosis. Do NOT re-activate until bounce source is found and fixed.
- This is the #1 deliverability risk; honoring it protects the primary domain.

**R3. When `unique_replied` becomes >0 (FIRST REPLY — this is the metric that actually moves stars):**
- **Same run, do these in order:** (a) read the reply content via the Apollo reply surface or via ken@ralphworkflow.com webmail; (b) classify the reply: positive (wants to learn more), neutral (info-sharing answer to your Mom-Test question), polite-pass; (c) append a full customer_discovery.jsonl entry per the Mom-Test schema (push/pull/anxiety/habit/implication); (d) upgrade ICP confidence in icp_findings.md §1 from MEDIUM to MEDIUM-HIGH (one real conversation validates the hypothesis); (e) treat the angle that produced the reply as the QUALITATIVE front-runner — its language informs the next sequence's copy, but it is NOT a statistical winner (winner claims need >=30 delivered/arm + >=3 replies per EXPERIMENT DESIGN STANDARDS); (f) build the NEXT sequence (V7) using that front-runner angle on the 3 expansion contacts (Joseph/Andrew/Felix) + reveal 4-6 more in the same ICP sub-segment.
- **Positive reply → respond within 24h from Ken's voice:** answer their points, continue the discovery thread, and make sure they have the repo link. Do NOT offer a call — per OUTREACH_COPY_CONTRACT.md §2 there is nothing to sell; a call happens only if THEY propose one (then log it and let it land in ken@ralphworkflow.com webmail). **If they want to go deeper (question, use case, feature idea, problem report): invite them to OPEN AN ISSUE on the repo (the surface their sequence linked) — owner-set escalation, public + durable + real repo engagement; never a call, there is no calendar link and nothing to demo.** Log the reply + your response to ledger.
- **Polite-pass → thank them, log the negative signal, do not retry** (the Mom-Test rule is "polite pass" is a valid answer, not a failure).
- This is the highest-leverage minute of the entire marketing program. Don't waste it on "I should also write a draft for someone else." The reply is the moment.

**R4. Day 3 milestone: if `unique_replied` is still 0 across all active sequences AND `unique_delivered` is >0:**
- The experiment is silent. Read the open signal from BOTH fields: `unique_opened_unfiltered` (raw, MPP-uncorrected, the real human-open count) and `unique_opened` (MPP-filtered, often 0 because Apple pre-fetches every send). If raw open rate >15% AND 0 replies, the angle is being read but not replied-to — the Mom-Test question is too hard, too off-topic, or the recipient tier is too senior (DevRels at observability orgs do NOT reply to cold strangers; OSPM maintainers are warmer). If raw open rate <10%, the subject line is failing. Either way, do NOT activate additional staged sequences yet — the in-flight data is not yet conclusive.
- **Concrete sharpen-copy action when raw-open>15%/0-reply at n≥10 (the exact pattern we are in on V3+V9 today: 27.3% raw open, 0 replies, 22 sent):** do ONE of the following this run, not all — (a) rewrite the subject line to under 36 chars and to a question, not a statement, (b) replace the Mom-Test question with a more specific, lower-effort ask ('how do you test your agent overnight?' beats 'how do you run multi-step agent tasks?'), (c) drop the recipient tier (Principal/Director DevRels are cold to strangers; Senior DevRels or DevRels at smaller orgs reply more), (d) rewrite the opening sentence to be a SPECIFIC OBSERVATION about the recipient's org (a 1-sentence per-contact opener 5x's reply rate vs generic). Log the sharpen to the ledger as `apollo_copy_sharpen` with the field changed + reason.
- **The #1 silent-phase action is FILLING THE ARMS (R8): top up each active arm toward 30 delivered with fresh verified on-ICP contacts, within the daily ramp cap.** Silence at n=2 is not data — it's an unfilled sample.
- Pre-stage: sharpen copy on the next staged sequence (the one R7 says is the next to activate if a current arm aborts) based on what you've learned (e.g. shorter subject, clearer question) and update the contact selections if the recipient profile taught you something.

**R5. Day 7 milestone — applies ONLY to FILLED arms. If an arm is underfilled (<30 delivered), the Day-7 action is FILL IT (R8/duty 4), not rotate it — you cannot conclude "this angle is silent" from a sample you never collected. When arms ARE filled (≥30 delivered each, real inboxes, not bounced) and still 0 replies after 7 days:**
- **Now activate the next 1-2 staged sequences** (oldest eligible `Ralph-AB-*` with all-verified contacts that are NOT already in flight) — `POST /emailer_campaigns/{id}/approve` for each. This is the explicit moment — the A/B structure (keep ~2 live variants) means the in-flight pair is "done" as the variant test, and the next staged pair takes their place. The 1-2 sends/day cap is preserved because the new pair are fresh sends and the old pair have already sent.
- Do NOT activate V5 (or any single-contact, last-resort sequence) yet. Single-contact sequences are a Day 14+ escalation.
- Log to ledger: "Day 7 escalation: in-flight pair produced 0/N replies, activating <seq_id>+<seq_id> as the next 2 live variants."

**R6. Day 14 milestone — same fill-first precondition as R5. If filled arms (≥30 delivered each) across two rotations are still reply-silent at Day 14:**
- **Now activate the last single-contact escalation sequence** (e.g. V5 / Drazen @ Nym). If 0/9 by Day 21, the email channel is wrong — pivot to LinkedIn DM (1:1, on-positioning, Mom-Test) for the next test. Update icp_findings.md §6 (Phase gate) to reflect the channel pivot decision.
- **Pre-stage the LinkedIn pivot ONCE this run if it has not been built yet (silent-experiment time is the right time):** draft a LinkedIn DM template using the same Mom-Test voice and ask ladder as the email sequences (Ken, founder-voice, ONE Mom-Test question, the GitHub repo link, polite-pass exit; NEVER a call/meeting ask — the same OUTREACH_COPY_CONTRACT applies). Save to `agents/marketing/drafts/<date>_linkedin_dm_template.md`. After that, only revise it if the positioning changes. If the email channel stalls at Day 14/21, the LinkedIn pivot is one-click, not a multi-hour rewrite.
- Update V6 design doc (Platform/DX engineer segment) to reflect whether it fires this cycle or stays staged.

**R7. The "ACTIVATE BEFORE YOU BUILD" rule vs the "keep ~2 live variants" rule — explicit resolution:**
- The activation floor (`apollo_activate_floor.py`, gate Layer 2.5) handles the literal "if any eligible staged sequence exists and 0 active sequences, activate one" case.
- The "keep ~2 live variants" rule means: do NOT activate V3+V4+V5 just because they exist, while an in-flight experiment is still running. Respect the experiment timeline (R4/R5/R6 above) for staged-sequence activation.
- Concretely: the right number of active sequences at any time is 2 (the current 2-variant A/B). 1 is too few (no A/B). 3+ dilutes signal and breaches the 1-2 sends/day cap.
- **ABORTED ≠ IN-FLIGHT.** If a previously active sequence has been aborted (status_reason=manual_pause/auto_pause with bounce, or via POST /emailer_campaigns/{id}/abort), it does NOT count toward the 2-active limit. In that case, the next eligible staged sequence (oldest `Ralph-AB-*` with all-verified contacts) MUST be activated to restore the 2-variant A/B — either by you (POST /emailer_campaigns/{id}/approve) or by letting the activation floor handle it. Do NOT defer to R5/R6 day-milestones when the A/B structure is broken by an abort. (R5/R6 govern activating a *third* live variant on top of two healthy ones, not restoring A/B after a sequence is lost.)

**R8. Anti-pattern guard — what NOT to do when the experiment is silent:**
- Do NOT build a V6 sequence pre-emptively while the in-flight 2-variant pair is still queueing/delivering.
- Do NOT write more drafts for new targets while the in-flight pair is running (the time is better spent sharpening existing copy and waiting for data).
- **FILLING THE ARMS IS ALWAYS IN SCOPE — a 2-contact arm is not an experiment, it's a placeholder.** What IS banned: enrichment that feeds nothing (revealing profiles that never get enrolled or engaged — credit spend as idle activity). What is MANDATORY every run: reveal + verify + ENROLL new on-ICP contacts into the ACTIVE arms (POST .../add_contact_ids with the mailbox id) until each arm reaches 30 delivered, under the deliverability ramp (≤10-16 new enrollments/day account-wide while mail-warming is off). Waiting for "Day 7 data" on n=2 is not patience, it's running no experiment at all — a human marketer fills the sample FIRST, then waits.
- The activity-theater guard is real. A run that ends with 0/N replies and a fresh draft for a 5th uncontacted person is failing the scorecard, not winning it. (Topping up an active arm is the opposite of that failure — it is the pipeline work the standards demand.)

**R9. Paused-on-dead-sequence contact recovery (defect D16).** A verified contact on a `paused` cc_status is RECOVERABLE, not lost — but Apollo's API does NOT support cross-sequence re-enrollment (`POST /emailer_campaigns/{id}/add_contact_ids` returns `skipped_contact_ids: {contact: "contacts_active_in_other_campaigns"}` for any contact already on any other campaign, even a paused/inactive one). When you discover a paused/aborted sequence holding verified ICP-fit contacts (the V1 case: 12 paused verified, 1 failed), the recovery paths are: (a) **reactivate the source sequence IN-PLACE with corrected copy** — the fastest path; you keep the verified-binding (mailbox per-contact, deliverability stats) and just swap the body via the campaign-level PATCH; (b) **re-reveal those exact verified emails** via POST /people/match (returns the same verified address) and enroll as fresh contacts in the new sequence (skips the API block, costs credits per reveal — only do this for high-value leads); (c) **headed-browser computer-use to move them via Apollo UI** — the only way to remove a contact from a campaign via the UI without losing the cc_status. NEVER treat paused contacts as terminal and walk away. Also: **before adding new contacts to a sequence, verify that sequence's `active:true` status** — adding to a sequence that is about to be aborted (the V1 top-up case: 5 contacts added at 21:11, sequence aborted at 21:22, all 5 auto-paused) is wasted credit spend. If the target sequence is `active:false`, do NOT add contacts to it; either activate it first (if it has at least 1 verified contact and is on-positioning) or use the clean-copy sequence instead.

**R10. POWERED-NULL ABORT-THEN-ROTATE (defect D49, evaluator 2026-06-10 22:08).** R5/R6 say "Day 7/14: activate the next 1-2 staged sequences." R5/R6 are SILENT on whether the powered-null pair is aborted FIRST. Combined with the activation floor's Guard J (POWERED-NULL brake blocks new cold activations while a powered-null arm exists), the system can deadlock: no new cold arms AND no abort of dead ones = continued burning of the warming mailbox for 0 outcome. The rule: **at the R5 milestone, ABORT the powered-null pair FIRST** (`POST /emailer_campaigns/{id}/abort` for each; verify `active:false` via a follow-up GET), THEN let the floor's next gate run activate the oldest eligible staged sequence (Guard J is now clear). Holding active arms at n>=30/0-reply past Day 7 burns the daily send budget on a dead channel — the same null-channel rule that says "don't pour a third powered arm into the same null channel" also says "don't keep the existing pair alive on that null channel." A powered-null arm that is still `active:true` at Day 7 is a defect to repair, not a patience test.

**R11. FIRE + CHECKBACK IS ATOMIC (defect D48, evaluator 2026-06-10 22:08).** Every time an H-item is fired (DRAFTED for owner approval, PENDING_OWNER, or executed for an already-approved-by-owner item), the same run that fires it MUST append a checkback entry to `logs/tactic_ledger.jsonl` with a clear `checkback: YYYY-MM-DD` (typically +3d for GitHub issues, +5d for HN, +7d for blog/awesome-list). The checkback is the marketer's obligation on the next run that observes the date — act on the maintainer's response (silent / replied-positive / replied-objection / merged-without-reply / closed) and either ship the amplification-when-merged action or append a new H-item. FIRE + APPEND CHECKBACK = ONE TURN. A 1-run gap is one run too many — the maintainer's reply sits un-read while the marketer runs another cycle. Same rule for H2 (Show HN) checkbacks: +5d after the fire is the first checkback, then +3d, +7d, +14d.
