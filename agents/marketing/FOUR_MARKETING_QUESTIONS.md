# The Four Fundamental Marketing Questions for RalphWorkflow

Every RalphWorkflow marketing message should answer these four questions clearly.

## 1. What is it?
Ralph Workflow is a free and open-source composable loop framework and AI orchestrator that runs the coding agents you already have on your own machine.

## 2. Who is it for?
It is for developers and technical teams who want to hand off real engineering work that is too big to babysit and too risky to trust blindly.

## 3. Why is it different / better?
Ralph Workflow is not just another chat window or one-shot code generator. It is fundamentally built on Ralph-loop ideas, but extends them into a composable loop framework: planning can loop, development iteration can loop, and the whole process can loop with explicit handoffs between phases. The result is repo-native orchestration that is simple at the core but strong enough for real software engineering work.

## 4. Why should I use it now?
Because it is free and open source, runs with the agents you already use on your own machine, and can help you wake up to an ambitious side project, major product chunk, or other serious engineering milestone that is ready to evaluate.

## Messaging rule
If a post, reply, page, README section, or asset does not answer these four questions either directly or implicitly, it is probably weak marketing and should be refined.

## Traffic rule
The marketing system's operational goal is to drive qualified traffic to the Ralph Workflow repositories:
- Codeberg first: https://codeberg.org/RalphWorkflow/Ralph-Workflow
- GitHub mirror second: https://github.com/Ralph-Workflow/Ralph-Workflow

Marketing activity that does not plausibly improve repo visits, repo inspection, or repo adoption should be deprioritized.

## Tight product phrasing to preserve
- free and open source
- composable loop framework
- orchestrates existing agents on your machine
- built on Ralph-loop ideas, but an improvement on Ralph
- planning loop + development iteration loop + overall process loop
- serious unattended work
- overnight while you sleep
- wake up to substantial, runnable, testable output
- whole project or large chunk of a project completed

## Guardrails for bad framing
- Do not frame Ralph Workflow as generic "Ralph" or the generic Ralph loop.
- Do not position it around small or narrow tasks.
- Do not lead with internal artifact plumbing or agent-to-agent handoff language.
- Do not lead with logs or diffs as the main proof; lead with what the software does now, what checks ran, and whether it is actually ready.
- Always remember Ralph Workflow depends on good software engineering practices: clear specs, meaningful tests, integration checks, executable software, and honest review discipline.
