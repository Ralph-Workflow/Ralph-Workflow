## What happened
Apollo gate at 04:00 CEST fired a Matrix FLEET-ALERT covering three loops:
- `run.py` — "recent error: Traceback ..."
- `run_posting.py` — "recent error: Traceback ..."
- `apollo_gate.sh` — same alert surfaced through the gate's own fleet monitor
  line `[fleet-monitor] 26 loops checked, 3 critical`

At 02:00 the same gate had said `2 critical` (run.py + run_posting.py only).
The 04:06 fleet_health.md regeneration said `0 critical`. Race + trim.

## Root cause (real, not false positive)
Cleanup commit `7d285cb9` (2026-06-08) deleted ~40 scripts as "dead theater
code" including:

- `log_janitor.py` (cron `0 4 * * 0` — still in crontab, silently failing since
  the cleanup; last real output 2026-06-07)
- `stale_artifact_watchdog.py` (cron `45 8 * * *` — same story)
- `distribution_lane_selector.py` — but `run.py` and
  `outcome_execution_board_runner.py` import it. The 02:00 and 04:00 cron
  windows caught it during a transient mid-write state where the
  working-tree untracked file was present but the package init couldn't
  re-export it yet. The 04:13 manual re-run after the trim had it solid.
- `channel_spidering_guard.py` — `run_posting.py` imports it as
  `agents.marketing.channel_spidering_guard`. Same story: untracked file,
  transient import failure.

A previous agent (or the cron race) had rewritten the apollo_gate_cron.log
trim header to claim the FLEET-ALERT lines were false positives. They were
real. Reverted that comment to honest wording as part of the fix.

## Lessons
- When a "cleanup" commit deletes cron-referenced scripts, the crontab
  silently fails — the fleet monitor only sees "no log written" and reports
  OK (because the log file isn't being touched). Future cleanups should
  parse the crontab first and refuse to remove anything still referenced.
- The 3-strike rule from AGENTS.md applies to silent script deletion too:
  this is the third time in June that script deletion has broken a cron
  chain (also: blind_monitor_replacement / reddit_monitor — those were
  the dead-job denylist in cron_integrity_test, removed honestly).
- The previous agent's "FLEET-ALERT lines removed (false positive)"
  comment was a doc-drift failure: trimming the log content for clarity
  is fine, rewriting the trim-header to lie about the cause is not. Fixed.
- Untracked working-tree files that the system depends on are a real
  failure mode for cleanups. The git add step is part of the fix, not
  optional cleanup.
