# APOLLO_DEFECT_CLASSES.md — the registry of named defect classes (append-only, never delete)

> WHY THIS EXISTS (2026-06-09): every defect class the evaluator polices was caught by the OWNER
> first — persona drift, call CTAs, n=2 "experiments", 2-contact arms, fixed-pool thinking, toy
> wave sizes, gate-metric inflation. The evaluator was checklist police: reliable on named classes,
> structurally blind to unnamed ones. This registry is the fix's memory half: every class ever
> caught, WITH the marketing-101 principle it violated, so (a) it is policed forever, (b) the
> ABSURDITY SWEEP (evaluator prompt) has a training set showing what "obvious" looks like.
> RULES: append-only (the wrapper alerts if this file shrinks). New entries come from the
> evaluator's absurdity sweep or the owner. Format: ID | name | the principle violated | how caught.

| D25 | Scorecard headline reports `unique_opened=0` while `unique_opened_unfiltered>0` (MPP-raw count hidden) | Two fields, one truth: Apollo's `unique_opened` is MPP-FILTERED (Apple Mail Privacy Protection pre-fetches every send and registers a 0-pixel open, which Apollo then credits to the recipient for tracking purposes only — but the actual HUMAN open is a separate signal tracked in `unique_opened_unfiltered`). Reading `unique_opened=0` and reporting "0 opens" in the headline hides the real open signal (V3 = 5 unfiltered opens on 16 delivered = 31% raw; V9 = 1 on 5 = 20% raw; the marketer or evaluator would have read "0 opens" and concluded the angle is invisible, when in fact the angle is being read at 28% raw). The fix: BOTH fields in the headline (`opens [unfiltered, MPP-raw; N tracked] (open rate X% raw / Y% tracked)`), the reply-rate denominator switched from `opens` to `sent` (replies/opens divides by MPP-inflated 0, producing 0% reply which is correct numerically but the wrong denominator for the standard reply-rate definition). **BINDING:** the scorecard's SEQUENCE OUTREACH headline MUST show BOTH `unique_opened_unfiltered` (raw) and `unique_opened` (tracked) so the next evaluator + marketer sees the actual read-rate and can run R4 (Day 3 silence protocol) against the right number, not the MPP-filtered 0. Replies remain the cleanest signal; the open-rate caveat (MPP-inflated) must be in the headline itself, not buried. | Evaluator (cold sweep), 2026-06-10 — caught after live API read of V3 (5 unfiltered opens, 0 tracked) + V9 (1 unfiltered, 0 tracked) on a scorecard headline that said "21 sent · 0 opens · 0 replies".
