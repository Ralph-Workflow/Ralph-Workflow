# Ralph Workflow Positioning Reference

This file is the durable product-definition reference for marketing and messaging.
Use it when writing README copy, landing-page language, social posts, comparisons, outreach, or review assets.

## What Ralph Workflow is

Ralph Workflow is a free and open-source composable loop framework and AI orchestrator.
It runs the coding agents you already use on your own machine.

It is fundamentally built on Ralph-loop ideas, but it is not just generic "Ralph" and should not be framed that way.
Ralph Workflow is an improvement on Ralph.

## Core conceptual framing

The key idea is simple at the center:
- loop planning
- loop development iteration
- loop the overall process
- keep explicit handoffs between phases
- keep the plan concrete enough to drive the next phase cleanly

That is why Ralph Workflow is more than just "AI orchestration."
It is composable orchestration built around loop structure.

A strong concise framing:
> Ralph Workflow takes the simple Ralph-loop idea and turns it into a composable loop framework for real software engineering.

Another durable framing:
> Ralph Workflow is the operating system for autonomous coding.

The full story should preserve all of these truths together:

- the core Ralph loop is simple
- that simple core composes into complex workflows
- that simplicity makes the system easier to configure, reason about, and extend
- Ralph Workflow is an AI agent orchestrator, not just another coding chat or thin wrapper
- the shipped default workflow is already very good for writing software
- users can start with the default workflow and build on top of it when needed

This should feel like **simple at the center, powerful in composition**.

## What it is for

Ralph Workflow is for big, ambitious, well-specified work.
A good first run should sound like:
- the fitness app you wanted to build
- a major product milestone
- a serious application slice
- a substantial engineering chunk with real acceptance criteria

It is not for:
- small tweaks
- narrow chores
- trivial one-hour edits
- vague product ideas with no spec

## How to evaluate it

The result comes first.
Do not lead with logs.
Do not lead with internal artifacts.
Do not lead with diff language alone.

Lead with:
- what the software does now
- what checks actually ran
- especially integration tests and other real guardrails
- whether the software is runnable, testable, and actually ready
- what a human can verify directly

Logs are fallback evidence only.
Diffs are useful supporting evidence, but not the primary promise.

## Engineering-practice dependency

Ralph Workflow depends on good software engineering practices.
It does not replace them.

Always preserve this framing:
- clear specs matter
- meaningful tests matter
- integration checks matter
- executable software matters
- honest review discipline matters

If a repo lacks those guardrails, Ralph should reflect that honestly instead of pretending the result is more trustworthy than it is.

- GitHub is the mirror.
- GitHub must never be treated as the write target for Ralph Workflow source updates.
- Update Codeberg; GitHub follows automatically as the downstream mirror.

Do not:
- collapse Ralph Workflow into generic "Ralph"
- describe it as if it were the generic Ralph loop
- position it around small or narrow tasks
- lead public docs with internal agent-to-agent or artifact-plumbing language
- lead with logs as if people should read giant logs first

- explain internal workflow plumbing before explaining why the product matters to the end user
- imply Ralph magically creates a good testing strategy when the repo does not have one
- treat GitHub like the primary repo when Codeberg is the source of truth

Do:
- frame it as an improvement on Ralph
- frame it as a composable loop framework
- keep the simple-core / serious-engineering contrast
- emphasize that the simple core is what makes more complex workflows easier to build and configure
- emphasize the strong default workflow for writing software
- emphasize that people can use the default first and extend it later
- emphasize ambitious, well-specified work
- emphasize runnable software and real checks
- emphasize result-first evaluation
- present Codeberg as primary and GitHub as the mirror

Top-level public docs should not lead with internal mechanics that do not matter to the evaluator or end user.

Usually keep these out of the top-level story unless directly necessary:

- artifacts as a featured concept
- internal handoff structure
- workflow plumbing

Top-level/public docs should instead lead with:

When writing public/product docs:

2. explain composition and orchestration second
3. explain the strong default workflow third
4. explain extension/configuration fourth

1: # Spec-Driven AI Agent: Why the Spec Still Matters
2: 
3: Ralph Workflow is a free and open-source AI agent orchestration system built around a simple Ralph-loop core.
4: That simple center composes into more complex workflows for substantial, well-specified software engineering work, and a written spec is what gives that stronger default workflow a real finish line.
5: 
6: If an agent keeps saying it is done before the work actually holds up, the problem is often not raw model capability.
7: The problem is that the task was never specific enough to verify honestly.
8: 
9: ## Why Ralph Workflow leans on specs
10: 
11: Ralph Workflow is designed for ambitious work that already deserves a clear target:
12: 
13: - a real feature slice
14: - a milestone with acceptance criteria
15: - a refactor with defined invariants
16: - a verification pass with concrete failure conditions
17: 
18: That is where a spec stops being ceremony and starts being operational.
19: It tells the workflow what done means.
20: It also gives the human a standard to review against afterward.
21: 
22: ## What a good spec changes
23: 
24: A serious spec helps in three places at once:
25: 
26: 1. **Planning** — the agent can make better choices when constraints are explicit.
27: 2. **Verification** — checks can be judged against something real instead of vibes.
28: 3. **Review** — the human can compare the result to the promised scope instead of reading tea leaves from a transcript.
29: 
30: Without that written target, even a strong model can produce work that sounds plausible while drifting away from what mattered.
31: 
32: ## Why this fits the default workflow
33: 
34: Ralph Workflow is not asking for giant design docs on every change.
35: It is asking you to use the workflow where ambiguity is expensive and a clear finish line matters.
36: That is one reason the default workflow works better on serious repo tasks than on tiny, vague chores.

2: 
3: Ralph Workflow is a free and open-source AI agent orchestration system built around a simple Ralph-loop core.

5: 
6: If you are looking for an unattended coding agent, the important question is not just whether a model can keep typing while you are away.
7: The important question is whether you can come back to software you can judge honestly.
8: 
9: ## Unattended should still mean accountable
10: 
11: A useful unattended workflow should leave you with:
12: 
13: - a clear task boundary
14: - software that actually runs or verifies better than before
15: - checks that tell you something real
16: - enough structure that you can understand what happened without replaying everything
17: 
18: That is the standard Ralph Workflow is trying to meet.
19: It is not promising magic independence from engineering discipline.
20: It is promising a stronger way to organize serious work.
21: 
22: ## When unattended use fits best
23: 
24: Ralph Workflow fits best when you have:
25: 
26: - a backlog task with a written finish line
27: - a repo with meaningful tests or validation
28: - enough scope that a simple chat session becomes awkward
29: - time to evaluate the result after the run
30: 
31: It fits worse when the task is vague, tiny, or dependent on constant mid-run steering.
32: 
33: ## Why the default workflow matters
34: 
35: Most people should not have to build an orchestration system before they can judge one.
36: You should be able to start with the shipped path, learn how it behaves on a real task, and extend it later only if you need to.

1: # What Good Ralph Workflow Output Looks Like
2: 
3: Ralph Workflow is a free and open-source AI agent orchestration system built around a simple Ralph-loop core.
4: This page is supporting proof for that composable workflow system and its strong default workflow, not the main product pitch.
5: 
6: Use this page after you already understand the workflow and want a review standard for the morning-after handoff.
7: Start with the product story and operator route first, then use this page to judge whether a run produced something worth trusting.
8: 
9: ## What to evaluate first
10: 
11: The first question is not whether the transcript sounds smart.
12: The first question is what the software does now.
13: 
14: Good output usually means:
15: 
16: - the task scope is recognizable from the result
17: - the repo is in a better state than before
18: - meaningful checks ran and their outcome is explicit
19: - the change can be reviewed against a real written spec
20: - the human can decide what to do next without reconstructing the whole run
21: 
22: ## Supporting evidence, in the right order
23: 
24: Use evidence in this order:
25: 
26: 1. **working behavior** — what changed in the software
27: 2. **real checks** — tests, integration checks, or other meaningful validation
28: 3. **written scope** — whether the result matches the promised task
29: 4. **supporting artifacts** — logs, diffs, or deeper traces if you need them
30: 
31: Logs and transcripts can be useful.
32: They just should not be the main promise.
33: 
34: ## What weak output looks like
35: 
36: Be skeptical when a run gives you:
37: 
38: - lots of narration but unclear product change
39: - a diff with no convincing checks
40: - a confident summary for a vague task
41: - artifacts that sound organized but do not make the result easier to judge
42: 
43: Ralph Workflow depends on real engineering guardrails.
44: If the repo does not have them, the honest outcome may be limited proof rather than full trust.
